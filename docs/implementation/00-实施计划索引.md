## 1. 本目录的职责

`docs/implementation/` 用于保存 RiskBench 的统一任务 Workflow、实施路线图、未来逐任务实现计划、施工进度账本和实施变更记录。

本目录不能替代：

- `AGENTS.md` 的仓库级治理；
- `docs/design/01–06` 的正式设计；
- Definitive MALF/Data 权威原文；
- 测试、命令输出、提交哈希和运行审计等实际证据。

若实施文档与正式设计冲突，必须停止施工并回到设计批准流程，不得通过修改实施计划改写领域语义。

## 4. 三层计划体系

### L1——实施路线图

回答“整个 v0.1 经过哪些里程碑、任务、依赖和门禁”。

它是全局结构，不保存每一步命令的实际执行结果。

### L2——任务实现计划

回答“某一个 Roadmap 任务具体怎样用 TDD 实施”。

统一命名：

```text
<任务ID>-<任务名称>-任务实现计划.md
```

统一存放：

```text
docs/implementation/tasks/
```

每个 Roadmap 任务在开始实施前都必须有一份独立计划，但采用逐任务滚动展开，不提前批量生成尚远的详细计划。

### L3——施工计划进度版

回答“目前实际执行到了哪个任务、哪个步骤，证据和提交是什么”。

全局文件固定为：

```text
docs/implementation/02-施工计划进度版.md
```

详细步骤和逐步勾选保存在对应任务实现计划；全局进度版只汇总任务级状态、当前 Step ID、阻塞、验证证据和提交哈希，禁止复制出第二套步骤真相。

## 5. 稳定编号

- 里程碑：`RB-M01`、`RB-M02`……；
- 任务：`RB-M01-T01`、`RB-M01-T02`……；
- 任务步骤：`RB-M01-T01-S01`、`RB-M01-T01-S02`……；
- 变更记录：`CR-0001`、`CR-0002`……；
- 门禁：`RB-GATE-003` 或 `RB-GATE-M01` 等。

编号一旦提交不得复用。被撤回的任务保留原编号并标记 `rejected`、`superseded` 或 `withdrawn`。

## 6. 统一任务状态

正常状态：

```text
planned
→ plan-drafting
→ waiting-for-plan-approval
→ approved-for-implementation
→ in-progress
→ verification
→ completed
```

异常状态：

```text
blocked
rejected
rolled-back
superseded
```

没有任务实现计划或计划未批准时，任务不得进入 `in-progress`。

## 8. Workflow 与三层计划的关系

统一流程标准：[`WORKFLOW-任务开发流程控制.md`](WORKFLOW-任务开发流程控制.md)。

```text
Workflow → Roadmap → 任务实现计划 → 施工计划进度版 → TDD/审计证据/Git 交付
```

所有任务实现计划必须声明遵循的 Workflow 版本；施工计划进度版只记录实际执行事实，不重新定义流程。
