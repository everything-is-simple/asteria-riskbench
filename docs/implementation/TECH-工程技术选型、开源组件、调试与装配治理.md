## 1. 目的与适用边界

RiskBench 不是通过“先把框架装上再摸索”推进。所有技术都必须能够被解释、验证、追踪、替换和回滚。

本文件解决以下问题：

- 某个技术为什么被选中；
- 某个开源项目什么时候可以进入系统；
- 组件如何在隔离环境中试炼；
- 试炼结果如何转化为能力卡；
- 主系统如何通过 Adapter 使用组件；
- 依赖如何锁定、升级和回滚；
- 组件失败时如何留下证据并退出；
- 前后端和构建链路如何避免互相污染；
- 哪些技术在 RiskBench v0.1 中明确禁止。

以下内容不属于本文件的职责：

- 改写 MALF v2.0 语义；
- 改变 G0–G3 门禁；
- 把研究数据提升为 operational 数据；
- 替代任务实现计划；
- 替代测试验收证据；
- 预先批准未来的隧道、小主机或云部署。

## 2. 技术选择总原则

### 2.1 优先级

技术选择按以下顺序评估：

```text
标准库与操作系统原生能力
  → 已被批准且低耦合的轻量依赖
  → 经隔离试炼验证的成熟开源库
  → 框架、数据库和复杂基础设施
  → 外部服务或远程托管能力
```

越靠后，举证和批准要求越高。不能因为某个技术流行、功能丰富或社区规模大，就跳过本地验证、许可证审查和失败边界记录。

### 2.2 RiskBench 的工程偏好

技术应尽量满足：

- 离线优先或本地可运行；
- 输入输出明确；
- 可用固定 fixture 重放；
- 不把业务语义隐藏在不可审计的黑盒中；
- 不绕过快照、门禁和谱系合同；
- 不要求后台常驻服务才能完成 v0.1 核心链路；
- 对 Windows 本地环境和未来小主机环境具有可解释的兼容性；
- 能在不迁移历史参考目录的前提下独立安装和验证；
- 失败时能够降级、替换或回到人工出口；
- 可以通过版本锁定和哈希复现。

### 2.3 不以“少写代码”为唯一目标

引入一个大框架如果导致：

- 运行边界不清；
- 依赖树不可控；
- 领域规则被框架约定吞没；
- 测试无法固定输入；
- Viewer 获得了不该有的写权限；
- 远程访问风险提前进入 v0.1；

即使它减少了代码量，也不能进入当前主路径。

## 3. 技术分级与批准条件

| 等级 | 技术类型 | 典型风险 | 最低准入条件 |
|---|---|---|---|
| T0 | 语言标准库、操作系统原生能力、仓库已有已批准能力 | 低 | 在任务计划中说明用途和边界 |
| T1 | 小型、低耦合、可替换的本地依赖 | 中低 | 技术选型记录、许可证、版本、最小 smoke test |
| T2 | 解析库、Web 框架、测试框架、序列化库等成熟开源组件 | 中 | 独立试炼、能力卡、输入输出和失败边界、任务计划批准 |
| T3 | 数据库、消息系统、后台服务、容器编排、远程服务或高耦合框架 | 高 | 独立设计、替代方案、运行与安全影响、专项门禁批准 |

规则：

1. 技术等级不是自动批准等级；
2. T0 也不能绕过当前任务门禁；
3. T2/T3 不得直接从“候选”变成“正式依赖”；
4. 任何技术跨等级使用，按较高等级治理；
5. v0.1 不因 Roadmap 中出现某项技术就视为已批准。

## 4. 技术选型记录

每次任务需要引入或改变技术时，任务实现计划必须包含技术选型记录，至少包括：

```text

候选技术和被选技术
要解决的问题
为什么标准库或现有能力不够
输入与输出
目标运行环境
许可证
固定版本和来源
直接依赖与传递依赖
离线行为
资源边界

```

如果记录不完整，任务不得进入 `approved-for-implementation`。

## 5. 开源组件准入流程

### 5.1 候选登记

候选组件只登记，不等于接入。登记必须记录官方仓库或官网、许可证、版本、维护状态、目标能力和初步风险。

不得仅凭搜索结果、博客、复制粘贴代码或旧系统中的安装状态认定组件可用。

### 5.2 隔离试炼

组件必须先在任务计划明确的隔离试炼目录中运行。试炼目录：

- 不得位于七个只读参考目录中；
- 不得被主系统运行时 import；
- 不得写入 `J:\asteria-trading-lab`、`J:\asteria-trading-labs-data`、`J:\asteria-trading-labs-Definitive-validated`、`J:\asteria-trading-labs-reference`、`J:\malf-data`、`J:\malf-system-history` 或 `J:\tdx_offline_Data`；
- 不得把试炼代码直接复制成正式业务代码；
- 必须能够删除并重新创建。

当前没有预先批准的 Composer 路径；具体试炼目录必须在任务实现计划中声明并经批准后使用。

### 5.3 Smoke test

最小 smoke test 必须验证：

- 组件可以在目标环境安装或启动；
- 最小输入能够产生可解释输出；
- 错误输入能够产生可识别失败；
- 不会静默访问外部网络或写入未授权目录；
- 版本和运行命令可重放；
- 退出、超时和资源异常行为可观察。

Smoke test 通过只代表“组件具备继续评估资格”，不代表可以接入 RiskBench。

### 5.4 能力卡

候选组件必须形成能力卡，至少包含：

| 字段 | 要求 |
|---|---|
| 组件名 | 官方名称和用途 |
| 官方来源 | URL、仓库或发行渠道 |
| 许可证 | 原文名称及兼容性判断 |
| 固定版本 | 版本号、锁文件或哈希 |
| 维护状态 | 发布频率、已知停更或风险 |
| 目标平台 | Windows、本地、小主机或其他 |
| 输入 | 类型、编码、大小和边界 |
| 输出 | 类型、稳定字段和错误形式 |
| 试炼目录 | 独立且可删除的路径 |
| Smoke test | 测试 ID 和证据 |
| 资源边界 | CPU、内存、磁盘、启动时间等 |
| 安全边界 | 网络、文件、执行和秘密访问 |
| 失败模式 | 可重试、可降级、需撤回或人工处理 |
| Adapter | 主系统封装边界 |
| 替代方案 | 组件不可用时的路径 |
| 状态 | `candidate` / `trial` / `approved` / `rejected` / `retired` |

能力卡必须和任务证据一起提交，不能只存在于聊天记录中。

## 6. Adapter 与主系统装配

RiskBench 主系统不得把第三方组件调用散落到领域逻辑和 Viewer 中。正式接入必须经过明确的 Adapter 或等价隔离边界：

```text
外部组件
    ↓
Adapter / 防腐层
    ↓
RiskBench 内部合同
    ↓
领域或构建流程
```

Adapter 必须负责：

- 外部输入输出转换；
- 版本差异隔离；
- 错误标准化；
- 超时、资源和取消行为；
- 许可证和来源注记；
- 不让外部组件定义 MALF 语义；
- 不让 Viewer 直接获得数据源、构建或发布权限。

Adapter 不得：

- 偷改领域规则；
- 把 `None + reason_codes` 改成默认中性值；
- 把 stale 数据伪装成 fresh；
- 绕过 G0–G3；
- 静默 fallback 到其他标的、周期、价格线或未来样本。

## 7. 依赖安装、锁定与变更

依赖安装只有在以下条件全部满足后才允许：

```text
Roadmap 已批准
→ 具体任务已登记
→ 任务实现计划已批准
→ 技术选型记录完成
→ 许可证和版本可审查
→ 安装范围已声明
→ 安装命令可重放
```

安装规则：

- 当前阶段未满足条件时不得安装任何业务依赖；
- 不允许运行时自动安装；
- 不允许未记录版本的浮动依赖；
- 不允许把依赖装入七个只读参考目录；
- 不允许通过环境变量或隐式全局安装掩盖项目依赖；
- 依赖锁定和许可证清单必须作为任务证据；
- 依赖变更必须说明新增、删除、升级或降级原因；
- 重大依赖变更必须有回归验证和回滚路径。

## 8. 调试流程

遇到技术、依赖或装配问题，必须按以下顺序处理：

```text
1. 固定问题描述和复现条件；
2. 固定输入、版本、命令和工作目录；
3. 建立最小可复现案例；
4. 在隔离试炼环境单独验证组件；
5. 检查 Adapter 的输入、输出和错误转换；
6. 检查 RiskBench 内部合同是否被破坏；
7. 保存成功和失败证据；
8. 决定修复、降级、替换、暂停或撤回；
9. 更新任务计划、能力卡和施工记录；
10. 经过 TDD 和门禁验证后才回到主系统。
```

禁止：

- 在主系统中无记录地反复试错；
- 把“本机能启动”写成“系统已验收”；
- 用真实行情偶然成功证明组件语义正确；
- 删除失败证据后只保留成功截图；
- 通过修改测试预期掩盖组件行为改变。

## 9. 升级、替换与撤回

所有依赖都必须有生命周期：

```text
candidate
→ trial
→ approved
→ active
→ deprecated
→ retired
```

升级或替换时必须：

- 建立变更记录；
- 保留旧版本和旧能力卡；
- 重新执行 smoke test 和受影响 TDD；
- 比较输入输出和失败边界；
- 验证快照、审计和回滚合同没有变化；
- 失败时恢复到最后一个已验证版本；
- 不得覆盖历史证据。

如果一个组件无法稳定重放、破坏离线边界、无法满足许可证要求或绕过系统合同，应标记为 `rejected` 或 `retired`，不得继续留在主路径。

## 10. v0.1 当前明确禁用的技术方向

在独立设计和批准前，v0.1 不启用：

- 后台 scheduler；
- 文件 watcher；
- 自动重算服务；
- 公网依赖或遥测；
- 写入式设置中心；
- 自动排名、推荐、交易建议；
- 订单、仓位、PnL 或资金系统；
- 为未来云部署提前引入消息队列、容器编排或分布式基础设施；
- 任何 AI Provider 或远程通知配置。

未来技术是否可用，必须通过新任务、新设计或独立门禁确认，不得因本文件存在而自动解禁。

## 11. 与任务 Workflow 的衔接

每个技术相关任务都必须同时遵循：

```text
AGENTS.md
→ docs/00-索引.md
→ docs/01-当前任务.md
→ docs/03-设计批准记录.md
→ docs/implementation/00-实施计划索引.md
→ docs/implementation/01-RiskBench-v0.1-实施路线图.md
→ docs/implementation/WORKFLOW-任务开发流程控制.md
→ 本技术治理文件
→ 当前任务实现计划
```

纯文档治理任务可以不执行代码和依赖步骤，但必须在任务记录中写明这些步骤为 `not-applicable`，而不是默认为已完成。

## 12. 完成定义

技术选型或组件装配任务只有在以下条件全部满足后才能标记 `completed`：

- 技术选型记录完整；
- 能力卡存在且版本明确；
- 许可证和来源已记录；
- smoke test 与受影响 TDD 证据已保存；
- Adapter 边界和失败行为已验证；
- 依赖锁定文件和变更记录已提交；
- 文档索引和当前任务已同步；
- Git 工作树、主线和远端状态已复验；
- 没有越过当前门禁或扩大授权范围。

## 13. 中断恢复

中断后不得依赖聊天上下文。恢复时必须：

1. 按仓库入口顺序读取文档；
2. 查找任务 ID、计划版本和当前步骤；
3. 检查能力卡、失败证据和 Git 状态；
4. 确认依赖和试炼目录是否处于批准范围；
5. 若状态不一致，停止装配并回到治理记录。
