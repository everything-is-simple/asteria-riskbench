# RiskBench 任务开发流程控制

- 文档 ID：`RB-WORKFLOW-001`
- 工作流版本：`riskbench-task-workflow-v0.2`
- 文档状态：`approved / governance-only`
- 批准门禁：`RB-GATE-004 / approved`
- 批准日期：`2026-07-20`
- supersedes：`riskbench-task-workflow-v0.1`
- 适用范围：Asteria RiskBench 全部治理、设计、实施、验证和发布任务

> 本 Workflow 只定义任务必须怎样被登记、计划、施工、验证、暂停、回滚和交付。它不单独批准依赖、技术栈、项目骨架或业务实现。

## 1. 核心制度

```text
里程碑批准
＋任务计划留痕
＋逐步施工记录
＋TDD
＋触碰红线立即暂停
```

目标：让任务在对话中断、执行者更换或失败后，仍能从仓库文件恢复真实状态，而不依赖聊天记忆。

## 2. 权威顺序

1. 用户已批准并提交的治理决策；
2. `AGENTS.md`；
3. Definitive MALF/Data 权威原文及登记哈希；
4. 设计 01–07；
5. 本 Workflow；
6. 当前 Roadmap；
7. 当前里程碑批准和特殊门禁；
8. 任务实现计划；
9. 施工计划进度版；
10. 代码、测试和运行证据；
11. 临时对话与未提交材料。

低层文件不得覆盖高层语义。冲突时停止施工，建立变更记录和 supersedes 裁决。

## 3. 强制开工入口

每次开始、恢复或切换任务必须按顺序读取：

1. `AGENTS.md`；
2. `docs/00-索引.md`；
3. `docs/01-当前任务.md`；
4. `docs/03-设计批准记录.md`；
5. 当前任务对应的设计、合同和权威来源；
6. `docs/implementation/00-实施计划索引.md`；
7. 当前 Roadmap；
8. 本 Workflow；
9. 当前任务实现计划；
10. `docs/implementation/02-施工计划进度版.md`。

MALF/Data 任务还必须读取 `AGENTS.md` 指定的 Definitive 原文并核验哈希。

入口检查必须确认：

- 当前里程碑已获批准；
- 任务 ID、范围、依赖和写入集合明确；
- 任务计划存在并通过里程碑定义的审查；
- 没有未处理的 redline、blocked、rollback 或冲突；
- 技术和依赖均来自已批准 selected 清单；
- 外部目录只读边界没有变化。

任一条件不满足，只允许治理修复或等待批准。

## 4. 里程碑与任务批准

### 4.1 里程碑批准

每个里程碑施工前必须明确批准目标、范围、非范围、依赖、selected 技术、验收矩阵、退出门禁和回滚边界。里程碑未批准，不得创建其业务实现或安装其依赖。

### 4.2 普通任务

已批准里程碑内的普通任务：

- 必须有稳定任务 ID；
- 必须有独立任务实现计划；
- 必须记录设计来源、文件写入集合、步骤、RED/GREEN/REFACTOR、验证和回滚；
- 必须进入施工进度账本；
- 可以按里程碑指定的审查角色批准，不默认要求用户对每个普通任务逐一点击批准。

若 Roadmap、里程碑或用户明确要求逐任务用户批准，仍按更严格要求执行。

### 4.3 特殊门禁与红线

特殊门禁和红线变化必须由用户明确批准，任何任务计划不能自行放宽。

## 5. 状态机

```text
planned
→ plan-drafting
→ ready-within-approved-milestone
→ in-progress
→ verification
→ completed
```

特殊状态：

```text
waiting-for-milestone-approval
waiting-for-special-gate
suspended-by-RB-GATE-004
blocked
rejected
rolled-back
superseded
```

`completed` 只在任务 DoD、文档、Git 和远端证据全部满足时使用。暂停、拒绝或 superseded 的任务保留历史，不删除、不复用 ID。

## 6. 任务实现计划最低合同

每份计划必须包含：

- 任务 ID、名称、Roadmap／里程碑来源；
- 当前状态和批准依据；
- 目标、非目标、依赖和假设；
- 权威设计条款；
- 精确写入路径；
- selected 技术与版本来源；
- 稳定 Step ID；
- 每一步的 RED、预期失败、最小 GREEN、回归范围；
- fixture、reason codes、审计和证据路径；
- 故障注入、停止条件、回滚；
- Definition of Ready / Done；
- Git 提交边界和远端验证；
- 中断恢复入口。

禁止用“实现功能”“完善测试”“调试一下”等不可验证描述代替步骤。

## 7. 标准施工流程

### Phase 0：登记与门禁确认

在 Roadmap 和进度账本登记任务，确认里程碑、特殊门禁、依赖、技术和写入范围。

### Phase 1：计划与独立审查

编写任务实现计划；审查范围膨胀、权威映射、依赖资格、失败行为、回滚和红线。未达到 ready 状态不得施工。

### Phase 2：隔离分支与环境

记录分支、基线提交、工作目录、运行时和依赖锁。禁止混入其他任务，禁止污染参考目录。

### Phase 3：TDD RED

先写与权威条款对应且能观察失败原因的测试。记录命令、退出码和失败摘要。不得由待测实现生成自己的 golden 预期。

### Phase 4：最小 GREEN

只实现当前 RED 所需最小行为。不得顺手实现后续里程碑、未批准依赖、交易功能、远程部署或 AI 扩权。

### Phase 5：REFACTOR

仅在测试通过后整理结构；不得改变批准语义、规则版本、存储边界或安全边界。行为变化必须走变更控制。

### Phase 6：分层验证

按任务适用范围执行：

- 单元与纯函数；
- 数据合同／状态合同；
- 领域语义；
- 门禁、谱系与审计；
- 持久化、事务、锁、备份／恢复；
- Viewer／API 边界；
- 响应式 viewport；
- 确定性重放、发布和回滚；
- 外部目录零写入和离线验证。

### Phase 7：失败、阻塞与回滚

失败时记录失败步骤、实际输出、已写文件、状态影响、回滚动作和恢复条件。禁止吞掉失败后继续。

### Phase 8：文档与施工记录收尾

同步当前任务、任务计划、进度账本、合同、批准／变更记录、证据路径和未完成边界。

### Phase 9：Git 交付

执行至少：

```powershell
git diff --check
git status --short
git diff --cached --check
```

只暂存任务精确范围；禁止习惯性 `git add -A`。提交后按批准边界推送，并验证本地 HEAD 与目标远端引用一致。

## 8. RiskBench 红线：触碰即暂停

以下变化必须停止当前任务、登记变更并重新获得用户批准：

1. 修改 MALF、adapter 或 fixed-point 精度合同；
2. 新增数据源、标的宇宙或价格线；
3. 扩大 AI 权限，使其计算风险、修改事实／声明或触发行动；
4. 引入券商账户、数量、成本、订单、成交、市场价值、PnL 或自动交易；
5. 将 Viewer 从 `127.0.0.1` 改为局域网（LAN）、公网、`0.0.0.0`，或引入隧道、小主机、云部署；
6. 引入未经 `RB-TECH-002` 或后续门禁 selected 的依赖；
7. 扩大已批准里程碑范围；
8. 把 `research_only` 升级为 `operational`；
9. 让 `var/` 清理触碰 `state/`，或降低用户资产备份／损坏停写要求；
10. 让 AI、UI 或存储层重算或改写 MALF 事实。

红线变化不能通过任务计划、重构或“临时调试”绕过。

## 9. 专属正确性纪律

- MALF/Data 必须遵循设计 03、04、06；
- `source_integer_fixed_point-v0.1` 与 adapter 非等价性必须有专项证据；
- `None` 必须有稳定 reason codes，禁止默认 neutral；
- 市场事实、用户声明、AI 解读必须可区分；
- 风险边界和距离必须由确定性规则计算；
- 用户资产使用 append-only revision；
- `var/` 可重建，`state/` 不可随意删除；
- Viewer 快照端点保持 GET/HEAD-only；
- 响应式测试不得被解释为真机网络访问。

## 10. 中断恢复协议

1. 重读强制入口；
2. 检查 Git 分支、HEAD、工作树和未跟踪文件；
3. 查看任务状态、最后完成 Step ID、证据和失败记录；
4. 检查进程、锁、staging、`current.json` 和 `state/` 状态（如适用）；
5. 验证已完成步骤，而不是凭聊天记忆认定；
6. 从下一个未完成步骤继续，或明确置为 blocked；
7. 禁止重复执行未知副作用的命令。

## 11. Definition of Ready

- 所属里程碑和特殊门禁已批准；
- 任务计划完整并通过规定审查；
- 依赖、selected 技术和版本明确；
- RED、fixture、预期失败和证据路径明确；
- 写入范围、外部只读边界和回滚明确；
- 无未处理红线或冲突。

## 12. Definition of Done

1. 批准范围内步骤全部真实执行；
2. RED 按预期失败，GREEN 最小通过，REFACTOR 后回归通过；
3. 权威条款到测试、fixture、结果和证据可追踪；
4. 故障注入和安全边界通过；
5. 文档、进度账本和状态同步；
6. 没有未批准依赖、范围膨胀或顺手实现；
7. `git diff --check` 和任务验证通过；
8. 提交边界清晰；
9. 按授权推送且本地 HEAD 与远端目标一致；
10. 未完成事项、限制和下一门禁明确。

任务完成不自动等于里程碑完成。里程碑出口必须独立复核。

## 13. Workflow 变更规则

修改本 Workflow 必须建立新版本、保留旧版本历史、登记 supersedes，不降低 MALF/Data 正确性、用户资产安全或远程访问红线，不用流程简化暗中扩大产品范围。
