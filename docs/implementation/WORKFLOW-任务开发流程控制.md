# RiskBench v0.1 任务开发流程控制

- 文档 ID：`RB-WORKFLOW-001`
- 工作流版本：`riskbench-task-workflow-v0.1`
- 文档状态：`approved / governance-only`
- 批准日期：`2026-07-19`
- 适用范围：Asteria RiskBench 仓库中的全部实施任务、治理任务和验证任务
- 权威位置：`docs/implementation/WORKFLOW-任务开发流程控制.md`
- 上游依据：`AGENTS.md`、`docs/00-索引.md`、六份正式设计分册、`docs/03-设计批准记录.md`
- 参考说明：本流程吸收 `I:\ai-studybuddy` 的“入口、计划、批准、验证、收尾”思想，但不复制其源码、原文、项目结构、命名体系或复杂度。

> 本文件是所有任务共同遵循的流程标准，不是 Roadmap，不是某个任务的实现计划，也不是施工进度日志。
>
> 本文件本身不授权安装依赖、建立业务项目骨架或实现业务代码。任务仍必须同时满足当前系统门禁、Roadmap 任务状态和该任务独立实现计划的批准条件。

## 1. 目的与不可替代性

RiskBench 的三层计划体系回答不同问题：

```text
Workflow
    ↓ 所有任务必须怎样做、经过哪些门禁
Roadmap
    ↓ v0.1 要做哪些里程碑、任务和依赖
任务实现计划
    ↓ 某一个任务具体怎样做、怎样验证和回滚
施工计划进度版
    ↓ 实际做到哪一步、留下了什么证据
```

本 Workflow 负责把每个任务变成可暂停、可审查、可重放、可回滚的执行单元，避免：

- 依赖聊天记忆推进任务；
- 没有批准计划就直接写代码；
- 计划步骤与实际施工步骤漂移；
- 用“测试通过”掩盖数据合同、门禁或文档治理缺失；
- 分支完成被误报为主线完成；
- 中断后凭记忆跳过步骤或重复执行有副作用的命令。

## 2. 权威级别与适用范围

### 2.1 权威级别

发生冲突时，按以下顺序处理：

1. 用户已批准并提交的治理决策；
2. `AGENTS.md` 的仓库级安全与写入约束；
3. Definitive MALF/Data 权威原文及其登记哈希；
4. 六份正式设计分册；
5. 本 Workflow；
6. Roadmap；
7. 具体任务实现计划；
8. 施工计划进度版；
9. 代码、测试输出和临时对话。

低层文件不得通过修改自身来覆盖高层语义。发现冲突时必须暂停任务，并建立治理变更或请求裁决。

### 2.2 所有任务均适用

以下任务同样必须遵循本 Workflow：

- 治理文档任务；
- 实施计划任务；
- 数据解析、聚合、MALF 和门禁任务；
- 发布、Viewer 和回退任务；
- 测试、验证、审计和修复任务；
- 文档、配置和运维边界调整任务。

纯文档任务可以省略不适用的代码步骤，但必须显式记录“适用 / 不适用”及理由，不能默认为跳过。

## 3. 强制入口顺序

每次开始、恢复或切换任务时，必须按顺序读取：

1. `AGENTS.md`；
2. `docs/00-索引.md`；
3. `docs/01-当前任务.md`；
4. `docs/03-设计批准记录.md`；
5. 当前任务对应的正式设计、合同和权威来源登记；
6. `docs/implementation/00-实施计划索引.md`；
7. `docs/implementation/01-RiskBench-v0.1-实施路线图.md`；
8. 本文件；
9. 当前任务的任务实现计划；
10. `docs/implementation/02-施工计划进度版.md`（产生后）。

MALF 或 Data 任务还必须按 `AGENTS.md` 指定的绝对路径读取 Definitive 原文，并核验 `docs/02-权威来源登记.md` 中的 SHA-256。

入口检查必须确认：

- 当前任务 ID 唯一且存在于 Roadmap 或批准的治理任务中；
- 当前总门禁允许该动作；
- 任务前置依赖已满足；
- 任务计划存在且版本明确；
- 计划批准状态允许当前阶段；
- 目标写入范围没有越过任务边界；
- 没有未处理的 `blocked`、`rejected`、`rolled-back` 或变更记录。

任一项不满足，任务只能停留在治理修复、证据整理或等待批准，不能进入业务施工。

## 4. 任务状态机

正常状态：

```text
planned
  → plan-drafting
  → waiting-for-plan-approval
  → approved-for-implementation
  → in-progress
  → verification
  → completed
```

异常状态：

```text
blocked
rejected
rolled-back
superseded
```

状态转换规则：

| 当前状态 | 允许进入 | 必要证据 |
|---|---|---|
| `planned` | `plan-drafting` | 任务已登记，Roadmap 来源明确 |
| `plan-drafting` | `waiting-for-plan-approval` | 任务实现计划完成，范围/非范围/验证/回滚齐全 |
| `waiting-for-plan-approval` | `approved-for-implementation` | 用户或指定批准者明确批准计划版本 |
| `approved-for-implementation` | `in-progress` | 开工入口检查通过，执行环境和分支已记录 |
| `in-progress` | `verification` | 计划步骤完成，证据和失败记录完整 |
| `verification` | `completed` | DoD、文档、Git、主线和远端验证全部满足 |
| 任意执行状态 | `blocked` | 阻塞原因、影响和下一动作已记录 |
| 任意执行状态 | `rolled-back` | 回滚对象、命令、结果和审计记录已记录 |

规则：

- 默认同时只能有一个任务处于 `in-progress`；需要并行时必须单独批准并声明文件所有权；
- 不得跳过 `waiting-for-plan-approval`；
- 不得把“代码写完”“分支已提交”或“本地测试通过”直接写成 `completed`；
- 状态只能由有证据的事实驱动，不能由口头进度驱动。

## 5. 任务计划与执行记录的职责分离

### 5.1 任务实现计划

统一存放于：

```text
docs/implementation/tasks/<任务ID>-<任务名称>-任务实现计划.md
```

它是“准备怎样做”的批准对象，至少包含：

- 任务 ID、版本、状态和 Roadmap 来源；
- 设计来源、权威条款和前置依赖；
- 目标、本期范围和明确非范围；
- 允许修改的文件和目录；
- 数据源、价格线、规则版本和隔离目录；
- Step ID、执行顺序、预期产物和停止条件；
- RED / GREEN / REFACTOR 策略；
- 验证矩阵、故障注入和 golden fixture；
- 回滚方案；
- Definition of Ready / Done；
- 计划批准记录和计划版本。

### 5.2 施工计划进度版

统一文件为：

```text
docs/implementation/02-施工计划进度版.md
```

它是“实际做到哪里”的全局账本，只汇总任务级状态、当前 Step ID、阻塞、验证证据和提交哈希。不得复制出第二套完整步骤真相。

每个任务实现计划中的步骤执行后，至少记录：

- Step ID；
- 实际开始/完成时间；
- 实际命令或人工动作；
- 修改文件；
- 结果与退出状态；
- 测试、日志或审计证据路径；
- 提交哈希；
- 是否偏离计划；
- 偏离时对应的 `CR-xxxx` 或阻塞记录。

计划文件定义预期步骤，进度版记录实际事实；两者冲突时暂停并修正，不静默覆盖。

## 6. 标准任务开发流程

### Phase 0：登记与范围确认

确认任务已经存在于 Roadmap，或有单独批准的治理任务 ID。登记：

- 任务 ID、里程碑和稳定名称；
- 设计来源和上游条款；
- 前置依赖；
- 目标交付物；
- 当前任务状态；
- 负责人或执行代理；
- 允许写入范围和明确非范围。

未登记任务不得创建业务文件。

### Phase 1：开工入口检查

执行第 3 节的入口顺序，检查当前门禁、计划版本、依赖和工作树。将检查结果写入任务计划或施工记录。

### Phase 2：编写任务实现计划

按第 5.1 节要求编写一份任务专属计划。计划必须把设计条款转成可验证的步骤，不得只写“实现某功能”。

每个步骤应至少有：

```text
Step ID
目的
前置条件
允许写入范围
动作/命令
预期结果
失败停止条件
证据位置
回滚方式
```

### Phase 3：独立审查与用户批准

计划完成后，先做一次独立审查，再提交批准。审查重点：

- 是否符合六份设计分册；
- 是否误把 research_only 当 operational；
- 是否引入未批准的价格线、规则或依赖；
- 是否遗漏坏数据、过期数据、相等比较和 `None + reason_codes`；
- 是否有可执行的 RED、验证和回滚步骤；
- 是否越过当前门禁。

计划未获明确批准，不得进入 `approved-for-implementation`，也不得实施。

### Phase 4：准备分支与隔离环境

计划获批后，才可根据任务需要：

- 从最新目标主线创建独立任务分支或 worktree；
- 安装计划明确授权的依赖；
- 创建计划明确授权的最小目录；
- 准备隔离测试数据和运行目录。

外部参考目录和 `J:\new_tdx64\vipdoc` 始终只读。任何会产生缓存、日志、数据库、锁文件或临时产物的命令，都必须将输出重定向到仓库自身受控目录或任务隔离目录。

### Phase 5：TDD RED

先建立失败测试、golden fixture 或结构验证，证明缺口存在。每条关键不变量必须追踪：

```text
权威条款
→ 测试 ID
→ fixture / 输入
→ 预期事件序列
→ 实际结果
→ 证据路径
```

RED 证据不得由待测实现自动生成。真实行情不能替代人工推导的 golden 预期。

### Phase 6：最小 GREEN

只实现使当前 RED 通过的最小范围。不得顺手实现后续任务、排名、推荐、Probability、operational 或其他未批准能力。

RiskBench v0.1 继续遵守：

- G0 失败整只标的拒绝，不跳过坏 bar；
- G1 过期或合同不符只能降级，不得伪造 fresh；
- G2 未形成字段保持 `None + reason_codes`；
- G3 lineage 不完整不得发布；
- raw_none 不得冒充 qfq_back；
- Viewer 不重算、不读取外部目录。

### Phase 7：REFACTOR 与局部回归

测试保持通过后才可整理结构。重构不得改变任务计划声明的行为、规则版本、快照格式或证据含义。任何行为变化都必须停下，提交变更记录并重新批准。

### Phase 8：分层验证

按任务计划声明的层级执行：

| 层级 | 必须证明的内容 |
|---|---|
| L0 | 文件、格式、文档结构和范围 |
| L1 | 单元、解析或结构不变量 |
| L2 | 数据合同、G0/G1/G2/G3 门禁 |
| L3 | Data → Core → Range 的纵向集成 |
| L4 | 快照、发布、Viewer、回退和审计 |
| L5 | 文档治理、diff、提交、主线和远端状态 |

任务计划必须声明适用层级；不适用的层级必须说明理由。

### Phase 9：独立验收与失败处理

独立检查：

- 结果是否可由输入和规则重放；
- 是否发生隐式 fallback、跨周期/跨标的借值或未来样本泄漏；
- 是否保留 `None + reason_codes`；
- 是否完整记录 lineage、规则版本和哈希；
- 是否越过任务文件边界；
- 是否有未登记的依赖或外部写入。

失败不得删除失败证据。根据情况进入 `blocked`、`rejected` 或 `rolled-back`，并记录：

- 失败步骤和时间；
- 输入、命令和结果；
- 影响范围；
- 是否污染了 staging 或运行目录；
- 下一动作和恢复条件。

### Phase 10：文档收尾

在标记完成前，必须同步：

- 任务实现计划的实际执行记录；
- `02-施工计划进度版.md` 的任务级状态；
- Roadmap 的任务状态（只写真实状态）；
- 当前任务文档；
- 验证证据索引；
- 变更记录和批准记录（如适用）；
- 未完成边界与后续门禁。

不得只因为代码和测试通过就宣称任务完成。

### Phase 11：Git 提交、合并与远端验证

提交前至少执行：

```text
git status --short --branch
git diff --check
任务计划声明的测试/验证命令
文档治理检查
```

任务分支完成不等于主线完成。完成判定必须包括：

1. 任务范围完成；
2. 任务验证矩阵通过；
3. 文档和施工记录已更新；
4. 提交已创建；
5. 已合入目标主线；
6. 主线重新验证通过；
7. 已推送远端；
8. `origin/main` 包含该提交。

交付说明必须写明分支、提交、是否合并、是否推送、验证命令、证据位置和未实现边界。

### Phase 12：完成或显式暂停

只有满足 Definition of Done 才能转为 `completed`。否则必须保留真实状态：

- `blocked`：等待外部条件或裁决；
- `rejected`：门禁或验证不通过；
- `rolled-back`：已回退到明确的已验证状态；
- `superseded`：被批准的新方案替代。

不得用“暂时完成”“基本完成”代替状态机状态。

## 7. RiskBench 专属硬门禁

任何任务都不得绕过以下规则：

- MALF 权威链为 `Core → Range → Lifespan → Probability → WaveProbabilitySnapshot`；
- v0.1 首个纵切只推进 Data/Core/Range；未形成字段保持 `None + reason_codes`；
- 永久撤回 27 桶、bucket、综合分、排名、推荐和交易建议；
- 当前 ETF adapter 为 `malf-v2.0-etf-tick-v0.1`，`raw_none`、`price_scale=1000`，原始整数严格比较；
- Pivot、Guard、Break 只用严格 `<` / `>`，相等不触发；
- TDX `.day` 只读解析，32 字节、`<5If2I`，坏数据整只拒绝；
- `research_only` 不得升级为 `verification_only` 或 `operational`；
- G0–G3 任一失败不得静默继续或发布；
- Viewer 只读不可变快照，不触发计算、不访问外部目录；
- `var/` 可删除、可重建且不进入 Git。

## 8. 变更控制

以下任何变化都必须先停止当前任务并建立 `CR-xxxx` 变更记录，重新获得批准：

- 新增、删除或重排 Roadmap 任务；
- 扩大任务范围或修改非目标；
- 改变接口、快照格式、规则版本、价格线或数据资格；
- 改变 Definition of Done、验证层级或回滚目标；
- 启用并行任务；
- 引入新依赖、外部服务、调度器或写入路径；
- 把 design-only 能力提前变成实现能力。

仅修正拼写、链接、证据路径或不改变语义的格式问题，可以在任务计划中记录后修订，但不得删除历史事实。

## 9. 中断恢复协议

任何任务中断后，恢复者必须：

1. 重新执行第 3 节入口顺序；
2. 读取任务实现计划和施工计划进度版；
3. 检查 `git status`、最近提交和未追踪文件；
4. 找到最后一个有命令、结果和证据的 Step ID；
5. 确认该步骤是否可安全重放；
6. 从下一个未完成步骤继续，或将任务置为 `blocked` 请求裁决。

禁止：

- 依靠聊天摘要判断文件是否已写入；
- 重跑可能造成覆盖、发布或外部写入的命令而不先检查状态；
- 重新解释已批准计划而不留下变更记录；
- 把未有证据的步骤勾为完成。

## 10. Definition of Ready / Done

### Definition of Ready

任务只有在以下条件满足后才可提交计划批准：

- 任务 ID 已登记且来源明确；
- 前置依赖和当前门禁满足；
- 权威条款、输入和写入范围明确；
- 本期范围、非范围和停止条件明确；
- RED fixture 或验证样例可独立推导；
- 验证命令、证据路径和回滚方法可执行；
- 未依赖未批准的后续能力。

### Definition of Done

任务只有在以下条件全部满足后才可标记完成：

1. 批准步骤和实际步骤记录完整；
2. RED 证据保留且符合预期；
3. GREEN 只实现批准范围；
4. REFACTOR 和回归验证通过；
5. 边界、坏输入和故障注入验证通过；
6. 无外部参考目录写入；
7. 规则版本、代码、测试和文档一致；
8. 施工进度版已更新；
9. 提交、验证命令和证据已登记；
10. 工作树符合任务计划要求；
11. 无未批准的顺手实现；
12. 目标里程碑门禁已通过，或任务明确声明不要求该门禁。

## 12. 本 Workflow 的修改规则

本文件是全局治理文档。修改必须：

- 说明修改原因、影响和上游依据；
