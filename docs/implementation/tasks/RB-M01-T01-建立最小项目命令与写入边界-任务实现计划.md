## 1. 任务目标

建立 RiskBench v0.1 的**最小、可测试、可追踪、边界明确的施工载体**，为后续 Data/Core/Range 纵切提供受控写入范围和可验证的命令合同。

本任务只解决：

- 仓库内部的最小项目边界；
- 构建端、发布端和 Viewer 的目录职责声明；
- 仓库写入与运行期 `var/` 写入边界；
- 未来命令入口的行为合同；
- 外部参考目录零写入的验证；
- 不依赖具体框架的 TDD/证据承载约定。

## 2. 不解决的问题

本任务不实现：

- TDX `.day` 解析、G0、G1 或任何 Data 逻辑；
- 周线/月线聚合；
- MALF adapter、Core、Range、Lifespan 或 Probability；
- G2/G3、快照内容、原子发布或回退；
- Viewer 页面、HTTP 服务或前后端具体框架；
- 开源组件安装、依赖锁定或技术栈最终选型；
- scheduler、watcher、遥测、远程通知；
- 隧道、小主机或云端部署；
- 立花、个人风险、AI Observer 或交易功能。

## 7. TDD 计划

### 7.1 RED 阶段

目标：先证明边界合同尚未满足，并保留可审查的失败证据。

- `RB-M01-T01-T01`：建立目录和写入边界的检查样例；
- `RB-M01-T01-T02`：建立命令入口行为的失败样例；
- `RB-M01-T01-T03`：建立“外部参考目录零写入”和“`var/` 可删除重建”的失败样例；
- `RB-M01-T01-T04`：建立未批准子系统不得出现的结构检查。

RED 预期：

- 检查因目标载体尚未存在而失败；
- 失败原因明确、稳定、可记录；
- 不使用待测实现自动生成预期；
- 不读取或修改参考目录。

### 7.2 GREEN 阶段

仅实现使上述 RED 样例通过的最小结构和边界：

- 创建经批准的最小项目目录；
- 添加最小命令入口或等价的受控检查入口；
- 明确构建端、发布端和 Viewer 的职责边界；
- 明确 `var/` 可删除、可重建且不入 Git；
- 使检查能够证明外部目录没有写入；
- 不添加任何 Data、MALF、门禁、发布或 UI 业务逻辑。

GREEN 阶段禁止：

- 为了通过结构检查预先创建后续里程碑的模块；
- 引入未经选择和批准的依赖；
- 把占位文件伪装成已实现能力；
- 用静态标记绕过真实边界检查。

### 7.3 REFACTOR 阶段

- 合并重复的边界检查；
- 使命令失败退出语义稳定；
- 使文档、检查、`.gitignore` 和施工记录一致；
- 不改变本计划批准的范围、输入、输出和停止条件；
- REFACTOR 后重跑全部 TDD 和治理检查。

## 10. 失败、停止与回滚

立即停止条件：

- 发现需要安装依赖才能继续；
- 需要选择未批准的具体技术或框架；
- 需要写入参考目录或 `J:\new_tdx64\vipdoc`；
- 需要创建 Data、MALF、发布或 Viewer 业务空壳；
- 发现 Roadmap、设计或 Workflow 条款冲突；
- 需要扩大精确写入范围；
- 任一边界检查无法产生可审计证据。

回滚方法：

1. 停止当前步骤，不删除失败证据；
2. 记录失败命令、输出、原因和影响；
3. 只撤销本任务已创建的仓库内受控文件；
4. 不触碰七个参考目录；
5. 将任务置为 `blocked` 或 `rolled-back`；
6. 如需改变范围，建立 `CR-xxxx` 并重新请求批准。

本任务不得使用“修复后继续”掩盖未审查的边界变化。
