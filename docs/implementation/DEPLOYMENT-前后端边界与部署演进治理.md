## 1. 稳定架构原则

RiskBench 的长期稳定边界是不可变快照合同，而不是某个前端框架、后端框架、服务器或云厂商。

```text
只读数据输入
    ↓
显式构建进程
    ↓
G0–G3 门禁
    ↓
不可变发布快照
    ↓
只读 Viewer / 只读访问层
```

以下部分应尽量可替换：

- 数据读取实现；
- MALF 计算实现；
- 构建命令；
- Viewer 技术；
- 服务器和部署位置；
- 快照物理存储。

以下合同不得因部署变化而改变：

- 数据来源身份、日期范围和 SHA-256 谱系；
- `None + reason_codes` 的降级语义；
- G0–G3 门禁；
- `malf-v2.0-etf-tick-v0.1` adapter 版本；
- 快照不可变和原子 `current.json` 切换；
- Viewer 不重算、不写外部目录；
- `operational` 在 v0.1 中禁用。

## 2. 逻辑组件职责

### 2.1 Data 输入层

负责：

- 只读访问已批准的数据源；
- 在构建时读取 TDX vipdoc；
- 记录源文件身份、大小、修改前后状态和 SHA-256；
- 将输入交给 G0 完整性检查。

不负责：

- 服务 Viewer；
- 写回 TDX；
- 接受浏览器写请求；
- 在浏览器刷新时重新读取数据。

### 2.2 构建与领域计算层

负责：

- 解析和聚合；
- 运行 MALF 适配层；
- 生成月、周、日不可变快照内容；
- 执行 G0–G3；
- 生成 lineage、reason_codes 和结果哈希。

不负责：

- 提供可写管理 API；
- 接收公网请求；
- 修改已发布快照；
- 将领域规则隐藏在 UI 中。

### 2.3 发布层

负责：

- 在 `var/staging/<run_id>` 构建；
- 仅在 G0–G3 全部完成后写入 `var/published/<snapshot_id>`；
- 原子更新 `current.json`；
- 保留发布和回退审计记录。

不负责：

- 静默覆盖 snapshot；
- 在构建失败时改变当前指针；
- 让 Viewer 直接访问 staging。

### 2.4 Viewer / 前端层

负责：

- 展示已经发布的不可变快照；
- 展示风险概览、标的详情和审计详情；
- 展示当前 profile、数据源、price_line、price_scale、规则版本和门禁状态；
- 对缺失字段展示已发布的 `None + reason_codes`。

不负责：

- 读取 TDX；
- 周期聚合；
- MALF 计算；
- 重新判断新鲜度；
- 补全或推断领域字段；
- 写入外部参考目录；
- 写入浏览器 localStorage 的机器路径或秘密；
- 触发构建、发布或回退。

## 3. 前后端分离决策

### 3.1 当前决策

```text
逻辑分离：立即执行；
物理分离：v0.1 不强制执行；
独立服务：只有在任务、设计和门禁批准后才允许。
```

因此 v0.1 可以采用单机形态：

```text
一个仓库
一个受控运行入口
一个本地 Viewer 端口
一个显式构建命令
```

但构建进程与 Viewer 必须具有不同的职责、输入和权限。是否使用某个具体前端或后端框架，不由本文件预先决定，必须遵循 `RB-TECH-001` 和具体任务计划。

### 3.2 稳定边界

构建端和 Viewer 之间只通过已发布快照合同传递结果：

```text
构建端 → 不可变快照 → Viewer
```

不允许：

```text
Viewer → TDX
Viewer → MALF 计算
Viewer → staging
Viewer → 发布命令
Viewer → 回退命令
```

## 4. 四阶段部署演进

### 阶段一：本地跑通

目标：证明治理、数据合同、Data/Core/Range 最小纵切、门禁和只读 Viewer 能在本地重放。

允许：

- `127.0.0.1` 本地绑定；
- 本地显式构建；
- 本地只读 Viewer；
- 本地 `var/staging`、`var/published`、审计和证据目录；
- 使用研究数据产生 `verification_only` 或 `research_only` 结果。

禁止：

- 自动 scheduler；
- 文件 watcher；
- 自动重算；
- 公网依赖；
- 遥测；
- 写入 TDX 或七个只读参考目录；
- 将 Viewer 作为构建服务使用。

阶段验收至少证明：

- 浏览器刷新不触发数据读取和重算；
- 构建失败不改变 `current.json`；
- Viewer 只能看到已发布快照；
- 快照 lineage、门禁和哈希可追溯。

### 阶段二：隧道只读访问

目标：在不改变领域和发布合同的前提下，让受控外部设备访问只读 Viewer。

在此阶段开始前必须单独建立并批准：

- 隧道访问与远程只读安全设计；
- 身份认证和访问控制；
- 隧道供应商及其网络边界评估；
- 断网、失效和凭证撤销方案；
- 访问日志和隐私边界；
- 公网暴露风险评估。

隧道只允许暴露：

```text
只读 Viewer；
只读快照访问；
必要的只读 API。
```

不得暴露：

```text
TDX 路径；
构建命令；
发布命令；
回退命令；
文件系统；
调试端点；
管理端点。
```

“能从手机打开页面”不等于远程访问安全验收通过。

### 阶段三：小主机运行

目标：将本地构建和 Viewer 放在受控的小主机上运行。

初始推荐形态仍可为单机：

```text
小主机
├── 只读 TDX 输入
├── 显式构建进程
├── var/staging
├── var/published
├── 审计和证据
└── 只读 Viewer
```

迁移前必须验证：

- 路径和权限合同；
- 时钟、交易日历和 evaluation_date；
- 磁盘空间和快照保留策略；
- 断电、重启和构建失败恢复；
- 备份、恢复和显式回退；
- 手机、电脑和平板的只读访问；
- 外部目录仍然只读。

小主机阶段不得因为硬件变化而自动开启 `operational`。

### 阶段四：云服务器评估

目标：在有明确需求、数据边界和安全设计后评估云端部署。

云端不能简单复制小主机目录。至少需要重新设计：

```text
数据输入边界
构建 Worker
快照存储
只读 API
Web Viewer
身份认证与授权
审计
监控
备份与恢复
秘密管理
成本与资源限制
```

云端部署必须作为新的设计和门禁，不是 v0.1 的自动下一步，也不能由本文件直接授权。

## 5. 网络与写入边界

在阶段一，Viewer 仅绑定 `127.0.0.1`，保持 GET/HEAD-only、离线优先、无遥测。

任何从本机外部可访问的阶段，都必须重新确认：

- 监听地址；
- 端口；
- 身份认证；
- 授权范围；
- 缓存和浏览器存储；
- 跨源策略；
- 错误信息是否泄露本地路径；
- 日志是否泄露秘密或个人信息；
- 隧道断开后的行为。

浏览器不得保存：

- 机器本地绝对路径；
- TDX 路径；
- 访问凭证；
- API 密钥；
- 任何可用于修改或回退快照的秘密。

触发文档必须成为对应任务的上游交付物，未完成时不得直接进入下一阶段。

## 7. 版本与回退原则

部署变化不得覆盖已有发布快照。任何阶段的发布和回退都必须遵循：

```text
已验证构建
→ G0–G3
→ 原子发布
→ 显式选择 snapshot_id
→ 原子切换 current.json
→ 独立审计记录
```

失败时：

- 不改变当前可用快照；
- 不静默回退；
- 不让 Viewer 读取 staging；
- 不删除失败证据；
- 不把网络可达性当作业务验收。

## 8. 阶段完成定义

任一部署阶段只有在以下条件满足后才能标记完成：

- 该阶段专项设计已批准；
- 入口、范围和非目标明确；
- 数据、构建、发布、Viewer 和网络边界有证据；
- 失败、断网、重启和回退路径已验证；
- 访问日志和隐私边界已确认（适用时）；
- 快照合同和 G0–G3 没有被削弱；
- 任务实现计划、施工进度、验证矩阵和 Git 证据完整；
- 用户批准的阶段门禁已满足。
