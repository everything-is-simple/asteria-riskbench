- 文档状态：`approved / extended-by-RB-GATE-004`
- 基线版本：`riskbench-design-v0.1 + product-realignment-v0.1`
- 首个 profile：`etf_raw_research_v0.1`
- 批准日期：`2026-07-19`
- 门禁合同：[`04-数据质量-分级门禁-新鲜度与审计.md`](04-数据质量-分级门禁-新鲜度与审计.md)
- 产品扩展：[`07-产品重定位与每日风险闭环.md`](07-产品重定位与每日风险闭环.md)

## 1. 产品定位

Asteria RiskBench v0.1 是本地个人风险工作台，不是交易终端、交易系统或自动化平台。

本分册原批准的“只读”边界专指**已发布市场快照 Viewer**。`RB-GATE-004` 增加了独立、受控、可写的用户风险工作区，使整个工作站可以保存相对暴露、风险边界、风险状态、笔记和可选 AI 解读，但不得让用户写入或 AI 改写市场事实。

交易员每日闭环及七个生命周期场景以设计 07 为准；v0.1 只实现核心交集，不为未进入 Roadmap 的场景创建空壳。

## 2. 运行方式

```text
显式构建快照
→ G0–G3 门禁
→ 原子发布
→ 本地只读 Viewer
```

构建进程与 Viewer 必须分离：

- 构建进程读取 TDX、聚合、计算、门禁和发布；
- Viewer 只读取已经发布的不可变 JSON；
- 浏览器刷新不得触发 TDX 读取、周期聚合、MALF 计算或快照重建；
- Viewer 不得访问或写入外部参考目录。

## 3. 市场快照 Viewer 的三个视图

### 3.1 风险概览

只展示固定标的清单，不进行动态排名：

- symbol；
- 最新数据日；
- evaluation date；
- usage/freshness；
- 月/周/日快照是否完整；
- 关键 reason_codes；
- 当前 profile 和规则版本。

### 3.2 标的详情

展示：

- 月、周、日分别的 Core、Range、Lifespan、Probability；
- 所有 None 对应的 reason_codes；
- 子快照 identity 和 `as_of_date`；
- price_line、price_scale、规则版本；
- 数据质量与门禁状态。

页面不得合成 27 桶、综合分或推荐。

### 3.3 审计详情

展示：

- run_id / snapshot_id；
- source identity 和 source hashes；
- 数据日期范围、bar 数量；
- parser/aggregation/MALF/adapter/profile 版本；
- G0–G3 结果；
- usage、freshness、reason_codes、warnings；
- 父快照和内容哈希；
- 显式回退审计事件。

审计详情展示审计事实，不修改审计内容。

## 4. 首屏强制信息

页面首屏必须显式展示：

- 绝对 `evaluation_date`；
- `latest_bar_date`；
- `research_only`；
- `stale_research_only`；
- `raw_none`；
- None 字段和原因；
- 当前快照身份。

禁止只使用颜色、tooltip 或二级页面隐藏这些限制。

## 5. Viewer 运行边界

| 边界 | v0.1 合同 |
|---|---|
| 网络绑定 | 只绑定 `127.0.0.1` |
| HTTP 方法 | GET/HEAD-only |
| 数据输入 | 只读已发布 snapshot JSON |
| 外部目录 | 不访问、不写入 |
| 网络依赖 | 离线优先，无公网依赖 |
| 遥测 | 无遥测 |
| 后台行为 | 无 scheduler、watcher、自动重算 |
| 页面写入 | 不写 snapshot、current pointer 或领域结果 |
| API | 不提供写 API |

POST、PUT、PATCH、DELETE 必须不可用。

## 6. v0.1 明确不建设

- 可写设置中心；
- 后台 scheduler；
- 文件 watcher；
- 自动重算；
- 动态排名和推荐；
- 买卖建议；
- 仓位、订单和 PnL；
- 市场快照 Viewer 内的可写 AI Provider 配置；
- 远程通知；
- 券商连接；
- 公网账户或云数据库。

首个 profile 和配置只读显示，不提供页面编辑能力。

## 7. profile 与配置可见性

页面必须只读显示：

- `profile_id=etf_raw_research_v0.1`；
- 逻辑数据源；
- 标的清单；
- `price_line=raw_none`；
- `price_scale=1000`；
- parser、aggregation、MALF 和 adapter 版本；
- evaluation date；
- freshness 与 usage。

机器本地绝对路径不得保存到浏览器 localStorage。市场快照 Viewer 不保存 AI Provider、API key 或通知配置；可选 AI 解读的本地秘密管理必须在 `RB-TECH-002` 和后续任务中单独设计。

## 8. 原子发布

目录合同：

```text
var/staging/<run_id>
→ 完整校验、G0–G3 和内容哈希
→ var/published/<snapshot_id>
→ 原子切换 current.json
```

规则：

- 在 staging 中完成所有构建；
- G0–G3 全部完成后才发布；
- 新构建失败不得改变 `current.json`；
- 同一 `snapshot_id` 永不覆盖；
- 发布目录一旦可见即必须完整；
- 半发布状态不得被 Viewer 读取；
- `var/` 不进入 Git，可删除并由同一输入和规则重建。

## 9. `current.json` 边界

`current.json` 只承担当前快照指针职责，不承载领域结果。

它的切换必须：

- 原子；
- 指向存在且已验证的 snapshot_id；
- 不因 Viewer 启动或刷新而改变；
- 不在构建失败时变化；
- 可通过显式审计回退更新。

Viewer 若发现当前指针、目标快照或内容哈希损坏，必须显示不可用，不得静默扫描旧版本。

## 10. 显式回退

禁止静默回退。

回退必须：

1. 用户或受控命令显式选择已验证的 snapshot_id；
2. 核验目标快照完整性和内容哈希；
3. 原子更新 `current.json`；
4. 生成独立回退审计事件；
5. 记录 from_snapshot、to_snapshot、原因、时间和执行身份。

回退不删除新快照，也不覆盖历史快照。

## 11. 写入边界

- 构建进程只写可重建派生数据 `var/`；
- 市场快照 Viewer 不写任何市场运行结果；
- 用户风险工作区只按批准合同写入 `state/`，且所有修改追加不可变 revision；
- `var/` 清理、发布和回退不得触碰 `state/`；
- 七个参考目录和 `J:\new_tdx64\vipdoc` 始终只读；
- 浏览器不得缓存机器秘密或未脱敏绝对路径；
- 派生日志只能位于 `var/`；用户资产、备份和写入锁只能位于 `state/`；
- `var/` 和 `state/` 都不得进入 Git。

## 12. 失败行为

| 失败 | 必须行为 |
|---|---|
| 构建失败 | 保留 current.json，不影响当前快照 |
| 门禁拒绝 | 不发布风险坐标，保留失败审计证据 |
| staging 中断 | 清理或隔离 staging，不产生半发布快照 |
| 当前快照损坏 | Viewer 显示不可用，不自动回退 |
| 回退目标损坏 | 拒绝回退并记录原因 |
| 市场快照 Viewer 的 HTTP 写请求 | 返回不允许，不触发 `var/` 或 `state/` 变化 |

## 13. 本分册验收条件

- Viewer 刷新不读取 TDX、不计算 MALF、不修改文件；
- 只存在风险概览、标的详情、审计详情三个视图；
- 首屏明确显示 stale、research_only、raw_none、绝对日期和 None 原因；
- Viewer 只绑定 127.0.0.1 且 GET/HEAD-only；
- 原子发布失败不改变 current.json；
- 同一 snapshot_id 不可覆盖；
- 损坏快照不会触发静默回退；
- 显式回退产生独立审计事件；
- v0.1 没有可写设置中心、后台调度或交易功能；Viewer 不提供 AI Provider 设置。可选 AI 解读的 Provider 只有在 `RB-TECH-002` 及后续任务批准后才能实现。

## 14. `RB-GATE-004` 运行边界扩展

### 14.1 Viewer 与工作站不再混为一体

- **市场快照 Viewer**：继续 GET/HEAD-only，只读 `var/published/`，不得写用户资产；
- **用户风险工作区**：允许通过后续批准的工作流写入 `state/`；
- **确定性风险服务**：验证 exposure bps、计算 cash、边界命中和距离；
- **AI 解读**：可选消费者，不是风险计算器，故障不得阻断确定性闭环。

任何实现都不得以“工作站可写”为理由让 Viewer 在刷新时读取 TDX、重算 MALF、修改已发布快照或访问外部目录。

### 14.2 `var/` 与 `state/` 永久分界

```text
var/     = staging / published / current.json，可删除、可重建
state/   = workspace data / backups / workbench.lock，不可再生用户资产
```

- 删除、重建或回退 `var/` 不得影响 `state/`；
- `state/` 损坏时停止写入，禁止静默创建空库；
- 破坏性迁移前必须备份；
- v0.1 只做本地备份／恢复，不做通用迁移包；
- 单写入进程必须持有 `state/workbench.lock` 的操作系统级独占锁；残留文件本身不构成永久锁。

### 14.3 响应式与网络红线

RiskBench v0.1 的手机、平板、桌面验收，仅指浏览器设备模拟、同机窗口尺寸调整或自动化 viewport 对响应式布局进行验证。它不表示允许手机或平板从另一台物理设备访问 RiskBench。

Viewer 继续只绑定 `127.0.0.1`。局域网真机访问、隧道穿透、小主机部署、公网访问和云端部署均属于后续远程部署专项，必须经过独立设计、安全审查和门禁批准，v0.1 不实现。

| 验收项 | v0.1 |
|---|---|
| `responsive-layout-test` | `allowed` |
| `same-machine-browser-test` | `allowed` |
| `physical-device-network-access` | `forbidden` |
| `LAN binding` | `forbidden` |
| `tunnel/public access` | `forbidden` |

不得以手机／平板验收为理由绑定 `0.0.0.0`、局域网 IP 或公网地址。

## 15. 扩展后的验收条件

除第 13 节原条件外，还必须验证：

- Viewer 写请求不能修改 `var/` 或 `state/`；
- 用户工作区写入不会修改已发布市场快照；
- `var/` 可删重建而 `state/` 内容和备份保持不变；
- 第二个写入进程拿不到 OS 独占锁时拒绝启动；
- 写入失败不留下半 revision；
- 同机 viewport 验收不产生外部网络请求；
- 服务仍只监听 `127.0.0.1`。

具体持久化、Web 服务和前端组件由 `RB-TECH-002` 决定；本分册不批准技术实现。
