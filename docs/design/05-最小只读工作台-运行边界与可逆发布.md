# 设计第 5 部分：最小只读工作台、运行边界与可逆发布

- 文档状态：`approved / design-only`
- 基线版本：`riskbench-design-v0.1`
- 首个 profile：`etf_raw_research_v0.1`
- 批准日期：`2026-07-19`
- 门禁合同：[`04-数据质量-分级门禁-新鲜度与审计.md`](04-数据质量-分级门禁-新鲜度与审计.md)

## 1. 产品定位

Asteria RiskBench v0.1 是本地个人风险工作台，不是交易终端、交易系统或自动化平台。

它只覆盖交易员生命周期中的：

- 观察市场；
- 形成可审计笔记。

盘前推演、错题复盘、交易计划、资金责任、立花库存、生存恢复和 AI 风险观察者继续保持 `design-only` 或 `not-started`。

## 2. 运行方式

```text
显式构建快照
→ G0–G3 门禁
→ 原子发布
→ 本地只读 Viewer
```

构建进程与 Viewer 必须分离：

- 构建进程读取 TDX、聚合、计算、门禁和发布；
- Viewer 只读取已经发布的不可变 JSON；
- 浏览器刷新不得触发 TDX 读取、周期聚合、MALF 计算或快照重建；
- Viewer 不得访问或写入外部参考目录。

## 3. v0.1 三个视图

### 3.1 风险概览

只展示固定标的清单，不进行动态排名：

- symbol；
- 最新数据日；
- evaluation date；
- usage/freshness；
- 月/周/日快照是否完整；
- 关键 reason_codes；
- 当前 profile 和规则版本。

### 3.2 标的详情

展示：

- 月、周、日分别的 Core、Range、Lifespan、Probability；
- 所有 None 对应的 reason_codes；
- 子快照 identity 和 `as_of_date`；
- price_line、price_scale、规则版本；
- 数据质量与门禁状态。

页面不得合成 27 桶、综合分或推荐。

### 3.3 审计详情

展示：

- run_id / snapshot_id；
- source identity 和 source hashes；
- 数据日期范围、bar 数量；
- parser/aggregation/MALF/adapter/profile 版本；
- G0–G3 结果；
- usage、freshness、reason_codes、warnings；
- 父快照和内容哈希；
- 显式回退审计事件。

审计详情展示审计事实，不修改审计内容。

## 4. 首屏强制信息

页面首屏必须显式展示：

- 绝对 `evaluation_date`；
- `latest_bar_date`；
- `research_only`；
- `stale_research_only`；
- `raw_none`；
- None 字段和原因；
- 当前快照身份。

禁止只使用颜色、tooltip 或二级页面隐藏这些限制。

## 5. Viewer 运行边界

| 边界 | v0.1 合同 |
|---|---|
| 网络绑定 | 只绑定 `127.0.0.1` |
| HTTP 方法 | GET/HEAD-only |
| 数据输入 | 只读已发布 snapshot JSON |
| 外部目录 | 不访问、不写入 |
| 网络依赖 | 离线优先，无公网依赖 |
| 遥测 | 无遥测 |
| 后台行为 | 无 scheduler、watcher、自动重算 |
| 页面写入 | 不写 snapshot、current pointer 或领域结果 |
| API | 不提供写 API |

POST、PUT、PATCH、DELETE 必须不可用。

## 6. v0.1 明确不建设

- 可写设置中心；
- 后台 scheduler；
- 文件 watcher；
- 自动重算；
- 动态排名和推荐；
- 买卖建议；
- 仓位、订单和 PnL；
- AI Provider 配置；
- 远程通知；
- 券商连接；
- 公网账户或云数据库。

首个 profile 和配置只读显示，不提供页面编辑能力。

## 7. profile 与配置可见性

页面必须只读显示：

- `profile_id=etf_raw_research_v0.1`；
- 逻辑数据源；
- 标的清单；
- `price_line=raw_none`；
- `price_scale=1000`；
- parser、aggregation、MALF 和 adapter 版本；
- evaluation date；
- freshness 与 usage。

机器本地绝对路径不得保存到浏览器 localStorage。未来 AI Provider、API key、通知配置不得进入 v0.1。

## 8. 原子发布

目录合同：

```text
var/staging/<run_id>
→ 完整校验、G0–G3 和内容哈希
→ var/published/<snapshot_id>
→ 原子切换 current.json
```

规则：

- 在 staging 中完成所有构建；
- G0–G3 全部完成后才发布；
- 新构建失败不得改变 `current.json`；
- 同一 `snapshot_id` 永不覆盖；
- 发布目录一旦可见即必须完整；
- 半发布状态不得被 Viewer 读取；
- `var/` 不进入 Git，可删除并由同一输入和规则重建。

## 9. `current.json` 边界

`current.json` 只承担当前快照指针职责，不承载领域结果。

它的切换必须：

- 原子；
- 指向存在且已验证的 snapshot_id；
- 不因 Viewer 启动或刷新而改变；
- 不在构建失败时变化；
- 可通过显式审计回退更新。

Viewer 若发现当前指针、目标快照或内容哈希损坏，必须显示不可用，不得静默扫描旧版本。

## 10. 显式回退

禁止静默回退。

回退必须：

1. 用户或受控命令显式选择已验证的 snapshot_id；
2. 核验目标快照完整性和内容哈希；
3. 原子更新 `current.json`；
4. 生成独立回退审计事件；
5. 记录 from_snapshot、to_snapshot、原因、时间和执行身份。

回退不删除新快照，也不覆盖历史快照。

## 11. 写入边界

- 构建进程只写仓库自身 `var/`；
- Viewer 不写任何运行结果；
- 七个参考目录和 `J:\new_tdx64\vipdoc` 始终只读；
- 浏览器不得缓存机器秘密或未脱敏绝对路径；
- 日志若未来存在，只能位于 `var/` 并纳入隐私和可重建策略；
- 所有运行状态都不得进入 Git。

## 12. 失败行为

| 失败 | 必须行为 |
|---|---|
| 构建失败 | 保留 current.json，不影响当前快照 |
| 门禁拒绝 | 不发布风险坐标，保留失败审计证据 |
| staging 中断 | 清理或隔离 staging，不产生半发布快照 |
| 当前快照损坏 | Viewer 显示不可用，不自动回退 |
| 回退目标损坏 | 拒绝回退并记录原因 |
| HTTP 写请求 | 返回不允许，不触发任何状态变化 |

## 13. 本分册验收条件

- Viewer 刷新不读取 TDX、不计算 MALF、不修改文件；
- 只存在风险概览、标的详情、审计详情三个视图；
- 首屏明确显示 stale、research_only、raw_none、绝对日期和 None 原因；
- Viewer 只绑定 127.0.0.1 且 GET/HEAD-only；
- 原子发布失败不改变 current.json；
- 同一 snapshot_id 不可覆盖；
- 损坏快照不会触发静默回退；
- 显式回退产生独立审计事件；
- v0.1 没有设置中心、后台调度、交易和 AI Provider 功能。
