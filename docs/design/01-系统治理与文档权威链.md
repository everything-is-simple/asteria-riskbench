## 1. 目的

本设计确保任何 AI、开发者或自动化工具在对话中断后，都能仅依靠仓库文件判断：

1. 什么是系统事实；
2. 什么已经获得批准；
3. 什么只是研究材料或临时草图；
4. 什么已经实现；
5. 什么已经通过验证证据；
6. 当前允许做什么、禁止做什么。

聊天记录、截图、临时 localhost 页面和旧仓库不能代替正式文档。

## 2. 系统宪章

Asteria RiskBench 是一个使用本地市场数据，为个人交易者发布可追溯 ETF 多周期市场结构与风险观察快照的本地只读工作台。

系统首先回答：

- 数据来自哪里、截止哪一天、是否过期；
- 月、周、日各自处于什么 MALF 结构状态；
- 哪些字段尚未形成以及原因；
- 当前结果经过了哪些规则、门禁和审计步骤。

系统永久不承担：

- 选股、排名、交易推荐和收益预测；
- 买卖建议、仓位、订单或 PnL；
- 自动交易；
- 用 AI 或 UI 重算、覆盖硬领域规则；
- 用综合分、27 桶或默认 neutral 掩盖未知。

## 3. 每次开工的强制入口

```text
AGENTS.md
→ docs/00-索引.md
→ docs/01-当前任务.md
→ 当前任务引用的正式设计、合同和决策记录
→ 涉及 MALF/Data 时读取 Definitive 原文并核验 SHA-256
```

入口文件缺失、互相冲突或当前任务不明确时，停止业务施工；只允许修复治理文档或请求用户裁决。

## 4. 文档权威链

发生冲突时，按下列顺序裁决，上层约束下层：

1. 用户明确批准并提交到仓库的治理决策；
2. 根目录 `AGENTS.md` 的施工与安全约束；
3. `docs/02-权威来源登记.md` 中登记并校验哈希的 Definitive 原文；
4. `docs/design/01–06` 中与问题直接相关的正式批准设计分册；
5. 聚合启动设计 `docs/superpowers/specs/2026-07-19-asteria-riskbench-bootstrap-design.md`；
6. `docs/03-设计批准记录.md`；
7. `docs/01-当前任务.md`；
8. 已批准的实施计划；
9. 代码、测试与验证证据；
10. 参考仓库的历史实现和测试；
11. 临时浏览器草图、聊天记录和未提交附件。

说明：

- MALF 领域语义始终以 Definitive v2.0 为最高语义权威；
- 代码只能实现已批准合同，不得反向改写产品边界；
- 正式分册是当前设计的引用入口，聚合文档保留审批形成过程和总体上下文；
- 同一事实不得在多处形成互相竞争的第二权威。

## 5. 七层文档治理模型

| 层级 | 文档职责 | 当前载体 |
|---:|---|---|
| 1 | 系统宪章：定位、永久非目标、安全和只读边界 | `AGENTS.md`、本分册 |
| 2 | 产品与领域地图：用户场景、MALF/立花/个人风险职责 | 分册 02、05；未来 PRD 可从正式设计派生 |
| 3 | 架构与数据合同：模块边界、字段语义、错误与降级 | 分册 02–05 |
| 4 | ADR：已接受的重要技术决策与理由 | 后续按决策建立独立 ADR |
| 5 | 路线图与任务账本：阶段、状态、计划索引 | `docs/01-当前任务.md`；未来 Roadmap/TodoList |
| 6 | 批准的实施计划：文件、测试、命令和回滚 | 下一门禁批准后创建 |
| 7 | 代码、测试与验证证据：实现事实 | 当前 `not-started` |

## 6. 分级门禁闭环

```text
任务登记
→ 设计 / ADR
→ 必要时人工批准
→ 实施计划
→ 测试驱动实现
→ 验证证据
→ 状态更新 / 提交
```

### 6.1 必须人工批准

- 产品范围和永久边界变化；
- MALF、立花或个人风险规则变化；
- 数据合同或持久化结构变化；
- 真实数据源、外部连接或写操作；
- AI 权限扩大；
- 进入实施计划、进入业务实现或启用 operational；
- 里程碑合并和远端发布策略变化。

### 6.2 在已批准计划内可连续执行

- 已列明测试的补充和缺陷修复；
- 不改变合同的内部重构；
- 文档同步和验证证据补充；
- 既定纵切中的最小实现；
- 本地提交与可逆实验。

超出批准计划即停止，不得以“顺手”“继续”或“上下文恢复”为由扩大范围。

## 7. 能力状态词汇

| 状态 | 含义 |
|---|---|
| `not-started` | 尚未开始设计或实现 |
| `research-only` | 只可用于研究材料，不表示当前或生产状态 |
| `design-only` | 已批准设计，但尚未实现 |
| `implemented` | 已存在实现，但未必通过全部验收 |
| `verified` | 已通过对应权威条款和追踪证据 |
| `rejected` | 未通过硬门禁，不得发布为有效结果 |

禁止把 `implemented` 写成 `verified`，禁止把 `research-only` 写成 `operational`。

## 8. 外部权威与只读参考边界

MALF 和 Data 权威文件、SHA-256、七个只读参考目录及数据源裁决，统一登记在 [`02-权威来源登记.md`](../02-权威来源登记.md)。

七个参考目录统一执行：

- 只读；
- 禁止修改、删除、移动和重命名；
- 禁止在其中生成缓存、日志、数据库、测试产物或锁文件；
- 禁止整体复制、目录迁移和代码搬运；
- 禁止新系统直接 import、软链接或运行时依赖旧仓库模块。

首个纵切的原始行情权威是 `J:\new_tdx64\vipdoc`；`J:\malf-data` 只做交叉核验。

## 9. 仓库写入与秘密边界

- 新系统只允许写入 `J:\asteria-riskbench` 自身受控文件；
- 运行期只允许写入被 Git 忽略的 `var/`；
- `.superpowers/` 是临时审阅材料，不进入 Git；
- `var/` 必须可删除、可重建；
- 秘密、机器本地路径配置和运行数据不得进入 Git；
- Viewer 不得写入外部目录、快照、指针或日志。

## 10. 冲突、变更与替代关系

治理文件修改必须说明：

1. 修改原因；
2. 影响范围；
3. 权威依据；
4. 是否替代旧决策；
5. 是否需要用户重新批准。

不得删除历史决策来制造一致。被替代设计必须保留，并通过 `supersedes` 或“已撤回”记录说明关系。

当前明确替代关系：

- “27 桶分类”已被分册 03 的月/周/日三个 MALF v2.0 不可变快照透明组合永久替代；
- 旧 Trading Lab 的 `Asteria-MALF-系统专属定义-v0.1.md` 只能作历史研究适配参考，不能提升为 RiskBench 权威。

## 11. 本分册验收条件

本设计只有在以下条件同时满足时才算被正确落实：

- 根目录存在 `AGENTS.md`；
- `docs/00-索引.md → docs/01-当前任务.md` 的入口可用；
- 外部权威路径和 SHA-256 有正式登记；
- 六份设计分册均在 Git 中可单独引用；
- 临时草图、运行数据和秘密未进入 Git；
- 当前任务明确列出允许动作、禁止动作和下一门禁；
- 任何能力都使用显式状态词汇，不以文案暗示已实现或已验证。
