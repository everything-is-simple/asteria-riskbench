# 设计第 4 部分：数据质量、分级门禁、新鲜度与审计

- 文档状态：`approved / design-only`
- 基线版本：`riskbench-design-v0.1`
- 批准日期：`2026-07-19`
- 数据权威登记：[`02-权威来源登记.md`](../02-权威来源登记.md)
- MALF 坐标合同：[`03-MALF-v2.0-透明多周期坐标.md`](03-MALF-v2.0-透明多周期坐标.md)

## 1. 目标

每个 RiskBench 结果都必须回答：

- 数据来自哪里；
- 数据截至哪一天；
- 解析是否完整；
- 使用了什么价格线和规则版本；
- 为什么可以发布、为什么只能降级、或为什么必须拒绝；
- 哪些字段未知以及稳定原因码；
- 是否能够用相同输入和规则确定性重建。

门禁只负责验证、分级和发布控制，不重新计算 MALF，也不把错误转换成 neutral。

## 2. 数据权威和裁决顺序

| 顺序 | 来源 | 用途 |
|---:|---|---|
| 1 | `J:\new_tdx64\vipdoc` | 首个纵切的原始 TDX `.day` 字段权威 |
| 2 | `J:\malf-data` | 交叉核验，不得覆盖原始记录 |
| 3 | `J:\tdx_offline_Data` | 历史/替代副本差异调查 |

差异处理：

1. 分别计算来源身份和 SHA-256；
2. 保留各自证据；
3. 报告差异，不静默择一；
4. 原始 `.day` 字段以 vipdoc 为准；
5. 不把交叉核验数据复制成新的未登记权威。

## 3. 四级门禁总览

```mermaid
flowchart LR
    G0["G0 输入完整性\n失败即 rejected"]
    G1["G1 数据合同与新鲜度\n决定用途等级"]
    G2["G2 模型完整性\n字段级 None + reason_codes"]
    G3["G3 谱系与发布\n缺审计即不发布"]
    G0 --> G1 --> G2 --> G3
```

门禁输出必须独立记录每一级：

- `gate_id`；
- `status`；
- `reason_codes`；
- `warnings`；
- `checked_at` 或确定性 evaluation context；
- 使用的合同和规则版本。

## 4. G0——输入完整性硬门禁

### 4.1 TDX 二进制合同

- `.day` 文件只读解析；
- 每条记录固定 `32` 字节；
- 文件长度必须可被 `32` 整除；
- 布局为 `<5If2I`；
- 最后两个字段按 unsigned 读取；
- 读取前后必须核验来源身份，读取期间源文件变化则整次拒绝。

### 4.2 日期不变量

- 日期可解析且合法；
- 严格递增；
- 不得重复；
- 不得乱序；
- 不得晚于显式 `evaluation_date`；
- 禁止依赖运行机器的“今天”隐式判定。

### 4.3 OHLCV/amount 不变量

对每条 bar：

- open、high、low、close 必须有效；
- 价格必须为正；
- `low <= open <= high`；
- `low <= close <= high`；
- volume、amount 不得非法或因 signed/unsigned 错误变负；
- 整数价格域不经 binary float。

### 4.4 故障处理

以下任一情况都必须整只标的 `rejected`：

- 31/33 字节尾部或其他截断；
- 重复日期、乱序日期；
- 未来日期；
- 非法 OHLC 或非正价格；
- 非法 volume/amount；
- 读取期间来源文件变化。

禁止跳过坏 bar 后继续计算，禁止产生“删除坏记录后的成功快照”。

### 4.5 固定回归

必须覆盖：

```text
symbol=512880
volume_raw=2793949440
```

不得错误解析为：

```text
-1501017856
```

## 5. G1——数据合同与新鲜度用途门禁

### 5.1 新鲜度合同

新鲜度以以下显式输入为准：

- `evaluation_date`；
- 经过批准的交易日历；
- `latest_bar_date`；
- 目标用途所需 `price_line`；
- 数据源和标的合同。

禁止使用“24 小时”“7 天”等任意自然日阈值猜测 fresh。

### 5.2 新鲜度状态

| 状态 | 含义 | 最大用途 |
|---|---|---|
| `fresh` | 最新 bar 等于交易日历推导的最近完整交易日，且价格线合格 | 仍受其他门禁限制 |
| `stale_research_only` | 数据结构有效但落后于最近完整交易日 | `research_only` |
| `freshness_unknown` | 交易日历或上下文不足，无法证明 fresh | `research_only` |
| `unsupported_price_line` | 当前价格线不满足目标领域合同 | `research_only` 或 `rejected` |

### 5.3 当前样本事实

- 五只 ETF 最后数据日：`2026-06-29`；
- evaluation date：`2026-07-19`；
- 当前 price line：`raw_none`；
- MALF Definitive 目标价格线：`qfq_back`。

因此当前样本只能：

```text
usage=research_only
freshness=stale_research_only
```

即使日期未来变新，`raw_none` 也不能自动升级为满足 `qfq_back` 的 operational 数据资格。

## 6. G2——模型完整性字段级降级

### 6.1 允许的 None

以下字段未形成或资格不足时，保持 None：

- Pivot；
- Wave；
- Range；
- Lifespan；
- Probability；
- peer rank；
- 多周期子快照；
- 组合 completeness。

### 6.2 稳定原因码

每个 None 必须附带机器可读 `reason_codes`。至少预留：

- `insufficient_history`；
- `uninitialized`；
- `range_not_formed`；
- `lifespan_not_available`；
- `insufficient_peer_sample`；
- `probability_not_verified`；
- `unsupported_price_line`；
- `freshness_unknown`；
- `stale_source`；
- `source_rejected`；
- `rule_version_missing`。

正式实施计划必须建立原因码登记表，禁止随意拼接自由文本代替稳定枚举。

### 6.3 永久禁止的 fallback

- 禁止跨标的借值；
- 禁止跨周期借值；
- 禁止跨 price_line 借值；
- 禁止使用未来样本；
- 禁止把历史实现结果填入当前字段；
- 禁止 neutral、flat、balanced 等默认值；
- UI 不得重新计算、补全或推断领域字段。

peer sample 少于 `30` 时，不生成相关 rank 或 Probability 字段。

## 7. G3——谱系与发布硬门禁

每个可发布快照至少包含：

### 7.1 运行身份

- `run_id`；
- `snapshot_id`；
- `produced_at`；
- `evaluation_date`。

### 7.2 来源身份

- 逻辑数据源；
- 机器内源路径或 source key；
- 文件 size、mtime；
- 来源 SHA-256；
- 首末数据日期；
- bar 数量；
- `latest_bar_date`。

机器绝对路径不得暴露到浏览器 localStorage；发布页面只显示经过合同定义的逻辑来源信息。

### 7.3 规则身份

- parser 版本；
- aggregation 版本；
- Core 版本；
- Range 版本；
- Lifespan 版本；
- Probability 版本；
- Service 版本；
- ETF adapter 版本；
- profile 版本。

缺少任何必要规则版本即禁止发布。

### 7.4 门禁和结果

- G0–G3 每级结果；
- usage；
- freshness；
- reason_codes；
- warnings；
- 父快照哈希；
- 规范化结果内容哈希。

缺少必要 lineage 的结果不得进入只读页面，也不得称为已发布快照。

## 8. 规范化序列化

发布内容必须支持确定性哈希：

- UTF-8；
- 键稳定排序；
- ISO-8601 日期时间；
- 原始整数价格保持整数；
- Decimal 使用规范字符串；
- 禁止 NaN 和 Infinity；
- reason_codes 使用稳定顺序；
- 相同输入、evaluation_date 和规则版本必须生成相同规范化结果和内容 SHA-256。

## 9. v0.1 用途等级

v0.1 只允许：

| usage | 条件 |
|---|---|
| `verification_only` | 合成 fixture 和确定性测试结果 |
| `research_only` | 通过 G0，但 stale、raw_none、freshness_unknown 或模型不完整 |
| `rejected` | 未通过硬门禁，不发布风险坐标 |

`operational` 明确禁用。未来启用必须提交独立设计、数据资格、测试证据和门禁审批。

## 10. 本分册验收条件

- G0 不允许跳过坏 bar；
- 新鲜度由交易日历和 evaluation_date 决定；
- stale、unknown 和不合格 price_line 明确降级；
- 所有未知字段使用 None + 稳定原因码；
- G3 缺 lineage 或规则版本时拒绝发布；
- 规范化序列化可确定性重放；
- v0.1 不出现 operational；
- 当前五只 ETF 只标记 research_only / stale_research_only。
