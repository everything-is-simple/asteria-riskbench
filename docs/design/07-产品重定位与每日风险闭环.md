## 1. 产品定位

Asteria RiskBench 是一套**风险管理为主、AI 为辅、以交易员每日场景驱动、只建设核心闭环的个人交易工作站**。

它不是：

- 自动交易系统；
- 选股、排名或收益预测系统；
- 券商账户、订单、成交、仓位或 PnL 系统；
- 由大语言模型计算或控制风险的系统；
- 为未来可能性提前堆叠框架、数据库和分布式组件的工程试验场。

### 1.1 五项产品原则

1. **风险管理优先**：所有界面、流程和解释首先帮助交易员识别事实、声明暴露、检查边界和保留责任记录；
2. **AI 只做辅助**：AI 只能解释、总结、提醒矛盾，不计算确定性风险数值，不修改事实或用户声明；
3. **只开发核心闭环**：没有进入每日风险闭环、没有通过里程碑门禁的能力不进入 v0.1；
4. **优先复用成熟免费开源组件**：复用必须先经过组件证据和 `RB-TECH-002`，不得把“开源免费”误解为“无条件增加依赖”；
5. **场景驱动产品**：功能必须对应交易员真实日常场景，而不是围绕框架、页面或数据库自我扩张。

## 2. 七个交易员生命周期场景

1. **观察市场**：确认数据日期、新鲜度、用途等级以及月／周／日市场结构事实；
2. **建立市场认知**：把市场事实、未知项、用户判断和后续观察点分开记录；
3. **盘前推演／模拟训练**：在不接真实订单的前提下演练边界触发与应对方案；
4. **复盘交易与决策**：复盘当时看见的事实、声明、行动和结果，不用事后信息改写当时记录；
5. **建立可执行方案**：把观察条件、风险边界、停止条件和行动预案写成可追踪声明；
6. **面对资金与责任**：明确相对暴露、现金缓冲、风险边界和本人责任，不把系统输出包装成建议；
7. **生存与恢复**：在风险触发、连续错误或状态恶化时执行暂停、复核和恢复流程。

v0.1 不要求一次实现七个场景的全部能力。v0.1 只实现支撑每日风险闭环的最小交集；其余场景保持设计入口，不创建空壳。

## 3. v0.1 每日风险闭环

```text
读取已发布市场快照
→ 核对数据日期、新鲜度和 usage
→ 查看月／周／日 MALF 事实
→ 声明五只 ETF 的相对暴露
→ 确定性计算风险边界与距离
→ 写入交易员风险笔记
→ 可选调用 AI 解释与提醒
→ 保存不可变 revision
→ 执行本地备份
```

闭环最低条件：

- 市场快照来自 G0–G3 后的已发布不可变结果；
- 明确显示 `evaluation_date`、`latest_bar_date`、freshness 和 usage；
- 市场事实、用户声明和 AI 解读在模型、界面和审计中可区分；
- 风险边界与距离由确定性规则计算；
- 每次用户修改新增 revision，不原地覆盖；
- 用户资产有可验证的本地备份／恢复路径；
- 删除 `var/` 不得删除任何用户资产。

## 4. 三层权威模型

### 4.1 市场事实

来源：TDX 权威输入、MALF、G0–G3、规则版本和确定性发布链。只能由确定性代码和已批准规则产生，包括数据日期、freshness、usage、月／周／日 MALF 结构、reason_codes、门禁结果和谱系。

### 4.2 用户声明

来源：交易员本人对相对暴露、风险边界、风险状态、笔记和行动预案的显式输入。必须带时间和 revision；它是“用户当时声明了什么”，不是系统测量，不得改造市场事实。

### 4.3 AI 解读

AI 只解释、总结和提醒矛盾：

- 必须标记为 interpretation；
- 不得写回市场事实或用户声明；
- 不得计算风险边界、边界距离或 MALF；
- 不得给出买卖、仓位或收益承诺；
- 故障、超时或禁用时不得阻断确定性工作流；
- 可以完整移除，而不影响市场事实和用户资产。

## 5. 统一术语表

| 术语 | 正式含义 | 明确不等于 |
|---|---|---|
| 风险 | 已定义不利条件、边界、暴露和未知项的组合 | 预测必然亏损、AI 情绪 |
| 暴露 | 用户按 `as_of` 声明的五只 ETF 相对资金占用比例 | 券商实盘仓位、市场价值 |
| 风险边界 | 用户或批准规则声明的确定性限制 | 买卖建议、安全承诺 |
| 边界距离 | 当前声明值与边界之间按确定性规则计算的差值 | AI 估计、未来收益空间 |
| 事实 | 可追溯来源和规则产生、可复核的记录 | 解释、推测、建议 |
| 解释 | 对事实或声明的含义说明 | 新事实、确定性结论 |
| 推测 | 尚无充分证据、明确标注不确定性的假设 | 市场事实 |
| 行动 | 交易员自己声明或执行的步骤 | 系统自动交易指令 |
| 未知 | 输入不足、资格不足或尚未声明 | 中性、零、安全 |
| 中性 | 经过明确规则得出的特定状态 | 未知、缺失、默认值 |
| 市场事实 | TDX、MALF、G0–G3 的确定性输出 | 用户意见、AI 文案 |
| 用户声明 | 交易员明确输入并留有 revision 的记录 | 系统测量、市场事实 |
| AI 解读 | AI 产生、可拔除的解释或提醒 | 市场事实、用户声明、风险计算 |
| 快照 | 给定来源、规则和 `as_of` 下不可变、可哈希的结果 | 可原地编辑的当前状态 |
| `research_only` | 只允许研究观察，不代表当前 operational 资格 | 可执行建议、生产可用 |
| `operational` | 未来经独立数据资格、安全设计、测试和门禁批准的用途等级 | v0.1 默认状态 |

永久硬约束：

```text
unknown ≠ neutral
AI 解读 ≠ 市场事实
用户声明 ≠ 系统测量
风险边界内 ≠ 安全
风险边界内 ≠ 建议持有
```

## 6. 相对暴露合同

首个闭环只允许 `510300`、`510500`、`159915`、`512880`、`513100` 五只 ETF。

1. 无杠杆、只做多；
2. 使用整数 `bps`，`10000 bps = 100%`；
3. 每只 ETF 的 `exposure_bps` 位于 `0..10000`；
4. `sum(etf_exposure_bps) <= 10000`；
5. 现金由确定性公式得到：

```text
cash_bps = 10000 - sum(etf_exposure_bps)
```

6. 每次声明必须有明确 `as_of`；
7. 不记录数量、成本、成交、市场价值、PnL 或券商账户；
8. 未声明时为 `unknown`，不得解释为零仓位、全现金或安全；
9. 相对暴露是用户声明，不是实盘账户测量。

## 7. 概念记录与不可变 revision

概念记录：

- `ExposureDeclaration`；
- `RiskBoundaryPolicy`；
- `RiskBoundaryAssessment`；
- `RiskStateDeclaration`；
- `TraderNote`；
- `AIInterpretation`。

共同要求：每次创建或修改生成新的不可变 revision；禁止原地覆盖；关联 record identity、前一 revision、创建时间、`as_of`、内容哈希和创建来源；`RiskBoundaryAssessment` 必须由确定性规则产生；`AIInterpretation` 单独保存；删除、撤回或修正必须用新 revision 表达。

本文只固定行为合同，不选择数据库、ORM 或序列化框架。

## 8. `var/` 与 `state/` 永久分界

### 8.1 `var/`：可重建派生数据

```text
var/
├── staging/
├── published/
└── current.json
```

`var/` 不进入 Git，可以删除，必须可从相同输入和规则重建；清理 `var/` 不得访问或删除 `state/`。

### 8.2 `state/`：用户不可再生资产

```text
state/
├── workspace data
├── backups/
└── workbench.lock
```

`state/` 保存用户声明、风险边界、风险状态、笔记、AI 解读、备份和写入锁：

- 不进入 Git，不得作为缓存随意删除；
- 不得被 `var/` 清理、重建或回退触碰；
- 损坏时必须停止写入并明确报错；
- 禁止因缺失或损坏而静默创建空库覆盖用户资产；
- 破坏性迁移前必须先生成并验证本地备份；
- v0.1 只实现本地备份／恢复；通用导出和跨机器迁移包延后。

## 9. 单写入进程合同

```text
单写入进程
＋事务性写入
＋追加新版本
＋禁止原地覆盖
```

1. 写入进程启动时对 `state/workbench.lock` 获取操作系统级独占锁；
2. 锁文件记录 PID、启动时间和实例 ID，仅用于诊断；
3. 第二个写入进程拿不到 OS 锁时拒绝启动；
4. 进程崩溃或退出后 OS 锁自动释放；
5. 残留锁文件本身不得永久阻塞，以 OS 锁状态为准；
6. 多标签页提示只属于 UX，不承担一致性责任；
7. 所有写入在事务内追加新 revision；
8. 写入失败不得留下半 revision 或覆盖旧值。

SQLite 是否采用、如何迁移和备份，必须由 `RB-TECH-002` 裁决；本门禁不选定数据库。

## 10. 确定性风险与 AI 权限边界

确定性代码负责验证 bps、不变量和总和，计算 `cash_bps`，按已批准 policy 计算边界命中与距离，输出 reason codes，保存 revision、哈希和审计。

AI 只能解释已有事实、声明和 assessment，总结笔记，提醒矛盾和未知项。AI 不得生成或绕过风险边界，不得替换确定性风险数值，不得把 unknown 改写成 neutral 或安全，不得修改 MALF、G0–G3、usage 或 freshness，不得产生买卖、仓位、订单或自动交易动作。

## 11. 响应式验收与远程访问红线

RiskBench v0.1 的手机、平板、桌面验收，仅指浏览器设备模拟、同机窗口尺寸调整或自动化 viewport。它不表示允许手机或平板从另一台物理设备访问。

Viewer 继续只绑定 `127.0.0.1`。局域网真机访问、隧道穿透、小主机部署、公网访问和云端部署属于后续专项，必须独立设计、安全审查和批准，v0.1 不实现。

| 能力 | v0.1 状态 |
|---|---|
| `responsive-layout-test` | `allowed` |
| `same-machine-browser-test` | `allowed` |
| `physical-device-network-access` | `forbidden` |
| `LAN binding` | `forbidden` |
| `tunnel/public access` | `forbidden` |

任何任务不得以手机／平板验收为理由改为 `0.0.0.0`、局域网 IP 或公网地址。
